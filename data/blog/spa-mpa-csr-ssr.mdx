프론트엔드를 공부하다보면 자주 접하는 용어인 SPA, MPA, CSR, SSR에 대해 알아보자.

## SPA

Single Page Application. 한 개(single)의 페이지로 이뤄진 어플리케이션을 의미한다.

**하나**의 HTML 페이지를 가지고 사용자에 요청에 따라 필요한 데이터만 서버에 요청해서 (JSON)을 통해 그 페이지를 갱신하는 방식이다.

리액트를 생각하면 쉽다.

SPA는 `CSR`방식으로 렌더링한다.

## CSR

Client Side Rendering.

사용자의 필요에 의해 요청된 부분만 응답받아 렌더링하는 방식이다.

최초로 실행할 때 초기 화면을 로드하기 위해 서버에 요청을 보내면, 서버는 모든 정적 리소스(HTML, CSS, JavaScript)를 전달한다.

이후, 사용자의 요청이 있으면 필요한 데이터만 서버에 요청해서 필요한 데이터만을 받아와 렌더링한다.

초기 화면을 보려면 모든 파일을 받아와야해서 초기 로딩 속도가 걸리고 빈 화면을 볼 수 있지만, 이후에는 한 페이지에서 사용자가 필요한 부분에서만 변경이 이루어지므로 속도가 빠르다.

![](/static/images/posts/csr.png)

### 장점

- 페이지의 리로딩이 없어 화면 전환이 자연스럽다.

- 서버 요청이 상대적으로 적어 서버 부담이 줄어든다.

- 전체 페이지를 렌더링할 필요가 없고 페이지 내에 일부만 바뀌기 때문에 속도가 빨라 좋은 사용자 경험을 제공한다.

### 단점

- 최초로 실행할 때 필요한 리소스들을 모두 받아오기 때문에 초기 구동에 시간이 걸린다는 단점이 있다.

- 어플리케이션 규모가 커지면 초기에 로딩하는 JS, CSS 번들 파일의 용량이 증가하고, 이는 초기 로딩 성능 저하의 원인이 된다.

- SEO에 취약하다: 검색 엔진의 bot은 파싱된 HTML만 대상으로 크롤링하기 때문에 SPA 경우에는 실제로 렌더링되는 데이터를 크롤링하기가 어렵다.

## MPA

Multi Page Application. SPA와 달리 여러 페이지로 구성된 전통적인 방식의 어플리케이션을 의미한다.

MPA에서 새로운 페이지로 이동할 때는 서버로부터 새로운 HTML을 받아오고 페이지에 대한 리소스들은 모두 다시 다운로드되는데, 이때 모든 페이지들에서 같은 일이 일어난다.

MPA는 `SSR`방식으로 렌더링한다.

## SSR

Server Side Rendering의 약자로, 서버로부터 완전한 HTML 파일을 받아와 페이지 전체를 렌더링하는 방식이다.

완전한 HTML이 먼저 받아와졌으므로 사용자는 페이지를 상대적으로 빠르게 볼 수 있다.

SSR 방식은 새로운 페이지를 요청할 때마다 새로고침되어 모든 정적 리소스가 다운로드 되고, 전체 페이지들이 다시 렌더링된다.

이때 **깜빡**거리는 현상이 나타난다.

SSR 방식은 일부 데이터만 갱신할 때도 전체 페이지를 다시 렌더링해야 하며, 정보의 변경이 많으면 그만큼 페이지 전체를 갱신하는 요청 횟수도 많아지기 때문에 비효율적이다.

또한, SSR 방식은 **보는 시간**(Time to view, TTV)과 **상호작용하는 시간**(Time to interact, TTI)에 차이가 존재할 수도 있다.

왜냐하면 HTML은 먼저 가져오기 때문에 화면에 보여질 준비가 된 상태이지만 브라우저가 자바스크립트를 (사용자가 interaction을 원할 때까지) 다운받지 못한 상태일 수도 있기 때문이다.

![](/static/images/posts/ssr.png)

### 장점

- 서버로부터 화면을 렌더링하는 필수 요소를 먼저 가져오기 때문에 초기 페이지 로딩 속도가 상대적으로 빠르다.

- 서버에서 완성된 HTML을 전달받기 때문에 SEO에 유리하다.

### 단점

- 서버가 과부하가 걸리기 쉽다.

- 프론트엔드와 백엔드가 서로 밀접하게 연관되어 있어 개발에 어려움을 겪을 수 있다.

- 페이지를 이동할 때 전체 페이지 렌더링으로 인해 깜빡거리는 현상이 일어난다. (좋지않은 UX)

- 전체 페이지 렌더링으로 인해 불필요한 데이터를 중복해서 로딩할 수도 있다.

- TTV와 TTI간 차이가 발생해 사용자에게 불편함을 줄 수 있다.
